# th06-sdl 安卓触控移植计划

## 目标

把东方夏夜祭这套“整张游戏画布触控 + 画布外左侧功能键 + 菜单触摸适配”的安卓输入思路，移植到 th06-sdl 里。

目标不是照搬 Unity UI，而是把它翻译成适合 th06-sdl 当前 SDL 输入架构的一套实现。

最终要达到的效果：

1. 安卓设备上可以用触摸正常游玩战斗。
2. 自机移动支持两种模式：
   - 虚拟摇杆
   - 触摸板
3. 功能键支持：
   - 射击
   - Bomb
   - 低速
   - 菜单 / 暂停
4. 菜单和对话不能只靠物理键，要支持触摸点选和基础方向导航。
5. 电脑版行为不被破坏，桌面平台仍继续走键盘 / 手柄路径。

---

## 一、现状判断

### 1. th06-sdl 当前已经具备的条件

从现有代码看，th06-sdl 已经有一个很适合承接安卓触控的核心点：

- 所有战斗输入最终会汇总成 `TouhouButton` 位标志。
- 这个统一入口就在 `Controller::GetInput()`。
- 手柄路径已经存在，说明“外部输入 -> 统一按钮状态”这件事本来就是这套架构的正常做法。

当前关键文件：

- [src/Controller.hpp](src/Controller.hpp)
- [src/Controller.cpp](src/Controller.cpp)
- [src/GameWindow.cpp](src/GameWindow.cpp)
- [src/thprac_gui_input.cpp](src/thprac_gui_input.cpp)

### 2. 当前缺口

缺的不是“按钮位定义”，而是下面几层：

1. 没有触摸事件状态缓存。
2. 没有“整张游戏画布触控 / 画布外按钮区”之间的触点仲裁。
3. 没有虚拟摇杆或触摸板逻辑。
4. 菜单输入目前基本还是纯键盘态读取。
5. 没有安卓触控布局配置和绘制层。

所以移植重点不是改战斗逻辑，而是补一层“触摸翻译器”。

## 二、移植原则

### 1. 不改战斗判定逻辑

战斗层仍然只读 `TouhouButton`。

也就是说：

- 整张游戏画布产生的移动触控最后仍然翻译成 `TH_BUTTON_UP / DOWN / LEFT / RIGHT`
- 射击键最后仍然翻译成 `TH_BUTTON_SHOOT`
- Bomb 最后仍然翻译成 `TH_BUTTON_BOMB`
- 低速最后仍然翻译成 `TH_BUTTON_FOCUS`
- 菜单最后仍然翻译成 `TH_BUTTON_MENU`

### 2. 触摸输入作为新增输入源，而不是替换键盘输入

最终输入结果应该是：

- 键盘输入
- 手柄输入
- 触摸输入

三者按位合并。

这样做有两个好处：

1. 安卓上支持触摸和手柄共存。
2. 桌面平台逻辑不需要分叉。

### 3. 菜单和战斗输入拆开处理

不能只做战斗触摸。

如果只给战斗加触摸：

- 菜单仍然无法正常操作
- 暂停、重开、选项页、弹窗都不完整

所以必须把：

- 战斗触摸层
- 菜单触摸层

作为两个子任务分别实现。

## 三、目标交互设计

## 1. 战斗场景

屏幕分成两大区：

### 整个游戏画布

负责方向控制。

这里要特别纠正：

- 不是左半区负责移动。
- 而是整个游戏画布都是方向触控区域。
- 玩家在画布内按下、拖动，都可以触发虚拟摇杆或触摸板解释。

支持两种模式：

1. 虚拟摇杆模式
   - 按下出现手柄中心
   - 手指拖动决定方向
   - 角色移动跟随方向

2. 触摸板模式
   - 手指在区域内滑动
   - 位移向量被量化成方向
   - 不强调摇杆视觉手柄

### 画布外左侧按钮列

负责功能按钮：

1. 射击
2. Bomb
3. 低速
4. 菜单 / 暂停

建议按钮默认布局：

- 射击：画布外左侧主按钮
- Bomb：射击按钮上方或左上
- 低速：射击按钮下方或中部
- 菜单：左上角独立小按钮

### 触点规则

必须采用“触点归属”机制：

- 第一个落在游戏画布内的触点，由方向控件接管。
- 落在画布外左侧按钮区的触点，只归对应按钮。
- 画布触控不能抢按钮区的触点。
- 按钮区也不能把已经归画布触控的触点重新解释为按钮。

这和东方夏夜祭的实现思路一致，也是防止误触的关键。

## 2. 菜单场景

菜单场景至少支持两种交互：

1. 直接点击
   - 点菜单项
   - 点确认按钮
   - 点返回按钮

2. 方向导航
   - 通过画布触控产生上下左右
   - 用射击键当确认
   - 用菜单键或 Bomb 当返回

建议优先实现“直接点选”，再补方向导航。

## 四、代码落点规划

## 阶段 1：新增触摸状态与统一输入桥

### 目标

先把触摸状态接入 `Controller::GetInput()`，让战斗层能读到触摸翻译后的按键位。

### 需要改动的文件

- [src/Controller.hpp](src/Controller.hpp)
- [src/Controller.cpp](src/Controller.cpp)
- [src/GameWindow.cpp](src/GameWindow.cpp)

### 具体工作

#### 1. 在 Controller 中新增安卓触摸运行时状态

建议新增：

- `TouchPointerState`
  - finger id
  - 当前坐标
  - 上一帧坐标
  - 首次坐标
  - 是否激活
  - 当前归属类型

- `AndroidTouchRuntimeState`
  - 方向触点
  - 功能键触点列表
  - 当前方向模式
  - 当前合成出来的 `TouhouButton`
  - 当前连续方向向量
  - 当前布局矩形

建议放在 `Controller.cpp` 的安卓分支里，和现有 `AndroidControllerDiagnostics` 类似分组管理。

#### 2. 在 GameWindow 事件循环中处理 SDL 触摸事件

当前 `GameWindow_ProcessEvents()` 需要补：

- `SDL_FINGERDOWN`
- `SDL_FINGERMOTION`
- `SDL_FINGERUP`

新增逻辑：

1. 把归一化坐标转换成游戏窗口坐标。
2. 根据当前场景判断这次触摸是：
   - 战斗输入
   - 菜单点击
3. 如果是战斗输入，再按“游戏画布内触控 / 画布外按钮区触控”分流。

#### 3. 在 Controller::GetInput() 中并入触摸结果

当前 `Controller::GetInput()` 已经会组合键盘和手柄。

这里新增一段：

- 读取 `AndroidTouchRuntimeState` 当前生成的按钮位。
- 与键盘、手柄结果按位或合并。

这样第一阶段结束后，战斗层不需要任何改动就能吃到触摸输入。

### 第一阶段验收标准

1. 安卓设备上能通过触摸让自机上下左右移动。
2. 射击、Bomb、低速、菜单四个按钮能分别点亮对应 `TouhouButton` 位。
3. 物理键盘 / 手柄仍然正常工作。

## 阶段 2：实现虚拟摇杆与触摸板两种方向模式

### 目标

把方向输入做成可切换模式，而不是把移动触控硬编码成固定小区域。

### 需要改动的文件

- [src/Controller.hpp](src/Controller.hpp)
- [src/Controller.cpp](src/Controller.cpp)
- 可能新增 [src/AndroidTouchInput.hpp](src/AndroidTouchInput.hpp)
- 可能新增 [src/AndroidTouchInput.cpp](src/AndroidTouchInput.cpp)

### 具体工作

#### 1. 抽出方向控件逻辑

建议新增两个函数：

- `UpdateVirtualJoystick()`
- `UpdateVirtualTouchPad()`

输入：

- 当前方向触点状态
- 当前屏幕布局信息

输出：

- `TouhouButton` 方向位
- 连续方向向量

#### 2. 虚拟摇杆模式

规则：

1. 方向触点首次按下时，记录中心点。
2. 当前点减去中心点，得到位移。
3. 位移限制在最大半径内。
4. 根据位移向量生成：
   - left / right / up / down
   - 可选模拟向量

#### 3. 触摸板模式

规则：

1. 方向触点首次按下时记录起始点或上一帧点。
2. 每一帧拿当前点减上一帧点，得到滑动方向。
3. 对向量长度做阈值过滤。
4. 把向量量化成 4 方向，必要时保留连续向量。

#### 4. 模式配置

建议新增配置项：

- `androidDirectionMode = Joystick | TouchPad`

默认建议：

- 安卓版默认摇杆
- 高级设置里允许切换到触摸板

### 第二阶段验收标准

1. 可以在配置里切换方向模式。
2. 两种模式都能稳定控制自机。
3. 游戏画布触控和画布外按钮区同时操作时不会互相误抢触点。

---

## 阶段 3：实现画布外按钮区与状态位映射

### 目标

把东方夏夜祭那种“功能按钮最终都写进统一输入状态”的思路翻译成 th06-sdl 的按钮位合成。

### 需要改动的文件

- [src/Controller.cpp](src/Controller.cpp)

### 具体工作

建立明确映射：

- 射击按钮 -> `TH_BUTTON_SHOOT`
- Bomb 按钮 -> `TH_BUTTON_BOMB`
- 低速按钮 -> `TH_BUTTON_FOCUS`
- 菜单按钮 -> `TH_BUTTON_MENU`

如果后面要做扩展功能，可再考虑：

- 长按射击时兼容 `TH_BUTTON_SKIP`
- 双按钮组合时兼容已有菜单快捷键逻辑

### 设计要求

1. 每个按钮都应支持按下、持续按住、抬起。
2. 方向输入与功能按钮输入必须可以并发。
3. 低速键必须允许在移动时持续保持。
4. 菜单键按下时必须能进入暂停 / 菜单逻辑。

### 第三阶段验收标准

1. 战斗中可双手同时操作：左手移动，右手射击 / 低速 / Bomb。
2. 连按和长按功能正确。
3. 没有明显的按钮黏连问题。

## 阶段 4：菜单触摸适配

### 目标

把当前几乎纯键盘化的 GUI 输入层补成“可点、可导航、可返回”。

### 需要改动的文件

- [src/thprac_gui_input.cpp](src/thprac_gui_input.cpp)
- 可能新增 [src/AndroidMenuInput.cpp](src/AndroidMenuInput.cpp)
- 可能新增 [src/AndroidMenuInput.hpp](src/AndroidMenuInput.hpp)

### 当前问题

现在 `thprac_gui_input.cpp` 基本靠 `SDL_GetKeyboardState()`。

这意味着：

- 触摸不会天然进入 GUI 输入系统。
- 菜单只能间接通过“把触摸翻译成假键盘输入”才能工作。

### 实施方案

建议分两步做：

#### 第一步：最小可用方案

把战斗触摸生成的：

- `TH_BUTTON_UP`
- `TH_BUTTON_DOWN`
- `TH_BUTTON_LEFT`
- `TH_BUTTON_RIGHT`
- `TH_BUTTON_SHOOT`
- `TH_BUTTON_MENU`

同步映射到 GUI 层对应的“确认 / 返回 / 方向”判断。

也就是说：

- 通过画布触控产生方向输入可导航菜单
- 射击键相当于确认
- 菜单键相当于返回

#### 第二步：直接点击方案

再给 GUI 增加“触摸热点点击”：

- 直接点菜单按钮
- 直接点选项框
- 直接点返回区域

这个阶段才需要更完整的矩形命中和 UI 布局数据。

### 第四阶段验收标准

1. 进入菜单后，不接物理键也能完成：
   - 选择项目
   - 确认
   - 返回
2. 暂停菜单可通过触摸正常恢复或退出。

## 阶段 5：触控布局与可视化

### 目标

让安卓触摸不仅能用，还要“看得见、能调位置”。

### 需要改动的文件

### 实施内容

#### 1. 绘制触控覆盖层

最少要绘制：

- 摇杆圆盘和手柄
- 射击 / Bomb / 低速 / 菜单按钮

2. 左右手切换后仍可正常游玩。

## 五、推荐实施顺序

为了降低风险，建议严格按下面顺序做：

1. 先做无 UI 的触摸输入注入
   - 不画按钮
   - 只验证能否通过触摸生成 `TouhouButton`

2. 再做方向模式
   - 先摇杆
   - 后触摸板

3. 再做功能按钮
   - 射击
   - 低速
   - Bomb
   - 菜单

4. 再做菜单导航兼容

5. 最后做可视化覆盖层、配置和美化

原因很简单：

- 如果先做 UI 画面，很容易陷入“看起来像能点，实际上底层没通”。
- 先把输入状态链路打通，后面只是表现层叠加。

## 六、与东方夏夜祭实现的对应关系

可以把移植关系理解成：

- `TouchDetector` -> th06-sdl 中的触摸状态缓存层
- `VirtualJoystick` / `VirtualTouchPad` -> th06-sdl 中的方向解释层
- `Panel_VirtualButton` -> th06-sdl 中的按钮到 `TouhouButton` 映射层
- `MyStandaloneInputModule` -> th06-sdl 中的菜单触摸兼容层

但不要机械照抄类结构。

因为 th06-sdl 当前不是 Unity UI 框架，它更适合做成：

- SDL 事件层
- 触摸翻译层
- 输入合成层
- 覆盖绘制层

四段式结构。

## 七、风险与注意事项

### 1. 菜单适配风险高于战斗适配

战斗只要能把按钮位喂进去，通常就能工作。

菜单则涉及：

- 焦点移动
- 确认
- 返回
- 文本输入或特殊弹窗

这部分很可能比战斗触摸更花时间。

### 2. 游戏画布触控和画布外按钮区的触点冲突是最容易出 bug 的点

如果不做触点归属，用户会遇到：

- 点射击时角色乱动
- 拖方向时误触 Bomb
- 多指时输入漂移

这是必须最优先处理的稳定性问题。

### 3. 桌面平台不能被安卓分支拖慢

所有新增逻辑都建议放在：

- `#ifdef __ANDROID__`
- 或运行时平台判断

避免桌面版每帧白跑触摸分支。

### 4. 触摸 UI 不要一开始做得太复杂

第一版目标应该是：

- 好用
- 稳定
- 不误触

而不是先追求复杂动画和华丽外观。

## 八、最小可交付版本定义

如果要切一个最小可交付版本，建议标准如下：

1. 安卓设备可进入游戏。
2. 战斗中可用触摸完成：
   - 移动
   - 射击
   - 低速
   - Bomb
   - 暂停
3. 暂停菜单可继续 / 返回。
4. 主菜单可上下选择并确认。
5. 键盘 / 手柄逻辑不回退。

只要做到这 5 条，就已经是一个能玩的安卓触摸版基础实现。

## 十、最终建议

最务实的推进方式是：

1. 先做战斗触摸输入注入。
2. 再做画布触控和画布外按钮区的仲裁。
3. 然后补菜单导航。
4. 最后才做 UI 覆盖层和布局配置。

如果按这个顺序推进，th06-sdl 恢复一套接近东方夏夜祭那种安卓操作兼容层，是可行的。
